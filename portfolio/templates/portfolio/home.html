Architecture Roadmap

The ongoing evolution of this platform from a static portfolio to a fully automated, AI-driven SaaS application.

Latest Update

Phase 8: SaaS Learning Portal & User Auth

The Goal: Wire up Django authentication, user ownership models, and build a secure Student Dashboard to consume unlocked curriculums.

The Struggle: The "SaaS Infinity Loop"—the dashboard linked to the marketing page, which linked back to the dashboard, with no actual private reading interface. Furthermore, Django's default auth routing caused 404 errors on login.

The Solution: Built a state-aware UI and a secure, two-column interactive Markdown study portal. Resolved routing issues via settings configuration, and utilized {% csrf_token %} protected POST requests for enrollment to prevent web crawlers from flooding the database with fake purchases.

Completed

Phase 7: Headless E-Commerce

The Goal: Implement a Composable Commerce architecture, decoupling the educational content engine from monetization and affiliate routing.

The Struggle: Merging data from disparate microservice-style databases (Core, Shop, Affiliates) into a single frontend view without hardcoding relationships, while safely handling missing database records.

The Solution: Built a dynamic Python bridge linking data via shared slugs. Fixed "Silent Template Failures" by tracing ORM reverse-relationships. Replaced clunky browser alerts with native JavaScript DOM toggles for a seamless, dark-mode showcase routing experience.

Completed

Phase 6: Unified Catalog UI

The Goal: Expose the massive database of AI-generated tech bootcamps to the public via a seamless, filterable frontend interface.

The Struggle: Rendering complex, highly relational data structures without overloading the database with N+1 query inefficiencies, while maintaining strict, aesthetic UI rules across varying content lengths.

The Solution: Built a dynamic Tailwind CSS Grid powered by Django's ORM parameters. Implemented category-based URL routing (?category=DS) that effortlessly filters the Master Database without requiring separate templates or hardcoded views.

Completed

Phase 5: Master AI Engine Refactor

The Goal: Eliminate severe technical debt by consolidating four disparate Python generation scripts into a single, highly modular pipeline.

The Struggle: The codebase was violating the DRY principle. Updating a model or Celery task required hunting down code across four separate 600-line files, drastically increasing bug risks.

The Solution: Executed a massive architectural refactor utilizing the Strategy Design Pattern. Reduced ~3,500 lines of procedural code into a sleek ~800-line Master Engine triggered dynamically via CLI flags. Normalized the database into a single TechBootcamp model.

Completed

Phase 4: Asynchronous AI

The Goal: Prevent the AI generation script from blocking the main Django server thread, allowing the web app to remain blazing fast.

The Struggle: Deploying the Celery worker to the AWS Linux environment resulted in immediate daemon crashes because the production virtual environment was missing core dependencies.

The Solution: Implemented a serverless Redis message broker via Upstash to handle the queue. Diagnosed the Linux systemd daemon crash logs, synced dependencies, and successfully detached the AI workload.

Completed

Phase 3: AWS Production Deployment

The Goal: Push the local architecture to a live AWS EC2 instance using a professional Git pipeline and serve it via Gunicorn and Nginx.

The Struggle: The site went live, but all CSS was missing. Nginx was returning a stubborn 404 Not Found for the Tailwind output file, despite the file existing. Overwriting Git history with build artifacts caused severe merge conflicts.

The Solution: Untangled the Git tree by ignoring the staticfiles artifact. Aligned the Nginx alias path with Django's STATIC_ROOT, instantly bringing the frontend online with proper Linux folder permissions.

Completed

Phase 2: Integrating the AI Generator

The Goal: Migrate a standalone Python/Ollama script into the Django ecosystem to securely generate tiered course data directly into the Neon.tech database.

The Struggle: Maintaining clean architecture. Dumping raw Python scripts into the root folder is an anti-pattern. We needed a way to execute the logic securely with access to Django's ORM.

The Solution: Refactored the script using strict Python Dataclasses and converted it into a native Django Management Command. This allowed terminal-based execution while preserving the separation of concerns.

Completed

Phase 1: Foundation

The Goal: Establish the core tech stack: Django backend, Neon.tech Serverless PostgreSQL database, and the newly released Tailwind V4 for styling.

The Struggle: Tailwind V4 introduced a critical infinite-loop bug during the watcher build process, constantly recompiling its own output file.

The Solution: Implemented strict file-path exclusions using the new @source directives in the CSS entry point, and surgically removed node_modules from Git tracking to keep the repository clean.
