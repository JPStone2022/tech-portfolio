Phase 6

The Unified Catalog UI & Dynamic Filtering

The Goal: Expose the massive database of AI-generated tech bootcamps to the public via a seamless, filterable frontend interface.

The Struggle: Rendering complex, highly relational data structures without overloading the database with N+1 query inefficiencies, while maintaining strict, aesthetic UI rules across varying content lengths.

The Solution: Built a dynamic Tailwind CSS Grid powered by Django's ORM query parameters. Implemented category-based URL routing (?category=DS) that effortlessly filters the Master Database without requiring separate templates or hardcoded views. The result is a blazing fast, composable UI that dynamically adapts to AI, Data Science, and Engineering content.

Phase 5

Architectural Refactor: The Master AI Engine

The Goal: Eliminate severe technical debt by consolidating four disparate, massive Python generation scripts (AI, Data Science, Backend, General Tech) into a single, highly modular pipeline.

The Struggle: The codebase was violating the DRY (Don't Repeat Yourself) principle. Updating a database model or a Celery task required hunting down and editing identical code across four separate 600-line files, drastically increasing the risk of deployment bugs.

The Solution: Executed a massive architectural refactor utilizing the Strategy Design Pattern. Built a single UniversalDayPlan dataclass and a master TOPIC_CONFIG map. This reduced ~3,500 lines of redundant, procedural code into a sleek ~800-line Master Engine that can be triggered dynamically via CLI flags (--topic DS). The database was similarly normalized, shifting from multiple app silos into one unified TechBootcamp model.

Phase 4

Asynchronous AI with Celery & Upstash Redis

The Goal: Prevent the AI generation script from blocking the main Django server thread, allowing the web app to remain blazing fast while heavy LLM processing happens in the background.

The Struggle: Deploying the Celery worker to the AWS Linux environment resulted in immediate daemon crashes because the production virtual environment was missing the core ollama dependencies required by the AI script.

The Solution: Implemented a serverless Redis message broker via Upstash to handle the queue. Diagnosed the Linux systemd daemon crash logs, synced the production dependencies, and successfully detached the AI workload into a non-blocking background process.

Phase 3

AWS Production Deployment & Nginx Routing

The Goal: Push the local architecture to a live AWS EC2 instance using a professional Git pipeline and serve it via Gunicorn and Nginx.

The Struggle: The site went live, but all CSS was missing. Nginx was returning a stubborn 404 Not Found for the Tailwind output file, despite the file existing on the server.

The Solution: After debugging Linux folder permissions and browser caches, a deep-dive directory traversal revealed the issue. Django was compiling static files into a folder named staticfiles, but Nginx was strictly configured to look in a folder named static. Aligning the Nginx alias path with Django's STATIC_ROOT instantly brought the frontend online.

Phase 2

Integrating the AI Generator

The Goal: Migrate a standalone Python/Ollama script into the Django ecosystem to securely generate tiered course data directly into the Neon.tech database.

The Struggle: Maintaining clean architecture. Dumping raw Python scripts into the root folder is an anti-pattern. We needed a way to execute the logic securely with access to Django's ORM.

The Solution: Refactored the script using strict Python Dataclasses and converted it into a native Django Management Command. This allowed terminal-based execution (python manage.py generate_programs) while perfectly preserving the separation of concerns.

Phase 1

Foundation: Neon.tech & Tailwind V4

The Goal: Establish the core tech stack: Django backend, Neon.tech Serverless PostgreSQL database, and the newly released Tailwind V4 for styling.

The Struggle: Tailwind V4 introduced a critical infinite-loop bug during the watcher build process, constantly recompiling its own output file.

The Solution: Implemented strict file-path exclusions using the new @source directives in the CSS entry point, and surgically removed node_modules from Git tracking via cached purging to keep the repository lightweight.
